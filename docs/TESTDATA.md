# テストデータでの網羅確認（#44）

現実的なデータを入れて全機能を通しで見るための手順書。
**ローカル専用**（`scripts/seed_testdata.py` は本番に届く経路を持たない。下の「安全性」参照）。

## 1. 起動

```bash
npx supabase start          # 初回は数分。10 コンテナが Up になる
make db-reset               # マイグレーション + seed でまっさらに
make testdata               # テストデータを投入

# .dev.vars が無いと /api/session が 500 になりログインできない
[ -f frontend/.dev.vars ] || cp frontend/.dev.vars.example frontend/.dev.vars

# VITE_ALLOW_CLOCK_OVERRIDE を渡さないと 24日の壁の ?now= が**黙って無視される**
cd frontend && VITE_ALLOW_CLOCK_OVERRIDE=true npm run build
npx wrangler pages dev dist --port 8788    # frontend/ から実行すること（functions/ を拾うため）
```

`http://127.0.0.1:8788` を開く。`DEV_BYPASS_EMAIL`（`frontend/.dev.vars`）が効くので
Cloudflare Access なしで「ゆるり」としてログイン済みになる。

終わったら `make testdata-remove` で撤去する。

> **ポートで詰まったら**: `curl :54321` が返らないのに DB コンテナだけ Up なら、Windows が
> ポートを予約している。`netsh.exe int ipv4 show excludedportrange protocol=tcp` を見て、
> 54320-54329 が除外範囲に入っていたら管理者 PowerShell で
> `net stop winnat && netsh int ipv4 add excludedportrange protocol=tcp startport=54320 numberofports=10 && net start winnat`。

## 4. 安全性（なぜ本番を汚さないか）

1. **接続先がローカルのコンテナに固定**されている（`docker exec supabase_db_yuruwollet psql`）。
   接続文字列を受け取らないので、**本番に届く経路が構造的に無い**。
2. **全行が固定 namespace からの決定的な UUIDv5**。`--remove` はその id 集合だけを消す。
   手で作った行や実データには当たらない（ミューテーションで検証済み）。
   ただし**チェックリストを実施すると DB 側が id を採番して作る行**があり、これは id で
   狙えないので条件で消す（テストデータに紐づくものだけ）:
   - サブスクの精算で生えた決済行（`subscription_id` がテストデータのサブスク）。
     FK が `on delete set null` なので、消さないと**普通の支出として台帳に居残る**。
   - 24 日の壁の `balance_checkpoints` と、確定で入る残高調整（`is_system_generated`）。
     消さないと**次回の壁が出なくなり**、手順が再現しない。テストデータのメンバー ×
     投入した月に限定している（それ以外の月の checkpoint には触らない）。
3. `profiles.opening_balance` だけは既存行の更新なので id で切り分けられない。
   投入時に元の値を `scripts/.testdata-opening-balance.json`（gitignore 済み）へ控え、
   撤去時に書き戻す。ただし**投入後にあなたが初期残高を変えていたら書き戻さない**
   （上のチェックリストは「初期残高を変えてみる」と指示しているので、素直に書き戻すと
   その編集を消してしまう）。控えが無いときも触らない（勝手に 0 にしない）。
5. 黙って壊れる 2 つは `--apply` 前に検査して止める:
   カテゴリが改名・削除されている（`insert…select` が 0 件になるだけでエラーが出ない）、
   同じ月に手で作った目標がある（`unique(member_id, period_month)` に当たって全体が落ちる）。
4. `is_system_generated`（残高調整）の行は作らない。**このスクリプトは `psql -U postgres` で
   繋ぐので RLS は効かず、作ろうと思えば作れてしまう**（アプリ側は RLS で禁止され RPC 経由のみ）。
   壁の挙動を壊さないよう**意図的に作らない**。24 日の壁は UI から確認する。
